# Dockerfile

Чтобы докеризировать наш проект, необходимо создать в корне проекта конфигурационный файл с названием `Dockerfile`.

В нём описывается, какую систему использовать в нашем контейнере и что сделать внутри контейнера после его создания, чтобы проект заработал
*По сути как инструкция к запуску нашего проекта, только не для других пользователей, а для докера*

На базе Dockerfile создаётся образ проекта, на базе образа создаётся контейнер/контейнеры нашего проекта

Вот основные команды по порядку, которые необходимо прописать в `Dockerfile`:

<hr>

```
FROM образ
```
Указываем образ, на базе которого будет строить свой (по сути - какую ОС с каким набором программ хотим). Образ автоматически скачается, если ещё не скачан
*Стандартный образ linux ubuntu+python: **python:3.14-slim***

<hr>

```
WORKDIR /my_project
```
Создаём внутри контейнера главную папку для нашего проекта. 
*Относительно этой папки будут работать и все последующие команды конфигурации*

<hr>

```
ENV ГЛОБАЛЬНАЯ_ПЕРЕМЕННАЯ=...
```
Задание глобальных переменных окружения. Они почти никогда не меняются, поэтому, учитывая послойность образов (о которой читайте ниже), задавать их следуюет наверху докерфайла. Они меняются даже реже, чем библиотеки в проекте
Глобальные переменные окружение - это те же переменные, которые мы указываем обычно в файле `.env`
Ими могут пользоваться (и пользуются) все программы внутри контейнера. В том числе и сам `Python` и базы данных и файловые системы и т.д.
С такими переменными из кода вы можете работать точно так же, как работаете с файлом `.env`
**Важное уточнение:** в `Dockerfile` лучше не прописывать свои переменные окружения (свои токены и прочее, что вы обычно пишите в файле `.env`). Так как они станут частью ораза и частью контейнера. 
У другого разработчика свои переменные окружения (свои токены и пр.). Лезть и менять всё это прямо в конфигурационном файле докера - плохая идея. Более того, образы часто загружаются на Dockerhub в публичный репозитрий. В котором каждому станут доступны ваши переменные окружения

<hr>

```
COPY что_копировать куда_копировать
```
Здесь `что_копировать` - какие файлы проекта вы хотите закинуть в контейнер. Не стоит кидать в контейнер тестовые файлы, файлы окружения, папки кэша и т.д. (их можно прописать в `.dockerignore`)
*Это нагружает контейнер, а смысл контейнера - быть легковесным*
А `куда_копировать` - в какую папку в контейнере мы хотим скопировать файлы 
**Пути работают относительно главной папки проекта, которую создавали выше**
Так что конструкция вида `COPY . .` скопирует всё из текущей папки терминала в текущую (главную) папку проекта контейнера
....

<hr>

#### Важный момент - послойность образов
В момент сборки образа, `Docker` смотрит, с какой команды начинаются изменения. Эта команда и все последующие просчитываются. А команды, идущие выше, результат которых не изменился, "тянутся" из прошлой версии

Например, базовый образ (1 команда) и название папки проекта внутри контейнера (2 команда) вообще вряд ли будут когда либо меняться. Они и логически стоят сверху. 

Это значит, что при повторной сборке докер не будет заново скачивать и "накатывать" образ и заново создавать папку внутри контейнера. Вы сэкономите и время, и ресурсы


Так что, правило простое: 
**Команды, результат которых будет меняться редко, нужно располагать выше. А команды, результат которых будет меняться часто, ниже в файле**

*Исходя из этого, выгоднее сначала скопировать файл `requirements.txt` и скачать библиотеки, ведь они меняются редко. А уже после копировать остальные файлы проекта*
<hr>

```
RUN команда терминала bash
```
`RUN` - запуск команд в терминале контейнера. 
*Сработает в момент сборки контейнера `docker build`*

<hr>

```
CMD ["аргумент1", "аргумент2", ...]
```
Например, `CMD ["python", "main.py"]`
`CMD` - тоже запуск команд в терминале контейнера, но сработает в момент запуска контейнера, а не в момент сборки 
*По факту, это значение по умолчанию. Его можно в любой момент переопределить. А если мы используем docker compose для поднятия программы из нескольких модулей, всё равно придётся переопределять команду запуска отдельно для каждого модуля
В таком случае здесь можно поставить команду запуска основного модуля*

<hr>


## Полный пример Dockerfile под Python проект
```
FROM python:3.14-slim

WORKDIR /my_super_project

# Устанавливаем для python рабочую корневую папку проекта, чтобы 
# нормально работали импорты (это необязательно - и без этого будет 
# работать, но так можно запускать контейнер любым удобным способом)
ENV PYTHONPATH=/my_project

COPY requirements.txt .

# Установка библиотек без создания папки кэша
# pip кэш прилично весит и отягощает контейнер
# Кэш помогает, если ты часто обновляешь библиотеки,
# а обычно этого уже не происходит на этапе докеризации
RUN pip install --no-cache-dir -r requirements.txt

# Копируем всё содержимое текущей директории 
# в рабочую папку контейнера
COPY . .


CMD ["python", "main.py"]
```
